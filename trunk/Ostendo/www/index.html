Ostendo - CORBA IIOP Message Analyzer

Corba messages are encoded using the binary IIOP protocol which yields small and efficiently transmittable and parseable messages. But these messages are not human readable. Using Ostendo (latin for "I reveal") IIOP messages can be displayed in an XML format.

Ostendo makes heavy use of JacORB's IDL parser and runtime to analyze IIOP messages.

Status

Development of Ostendo is based on this IDL interface server.idl, there are unit tests based on the interface. New features are first put into the IDL, then test Client and Server implementations are enhanced and run. Using the resulting messages the Ostendo implementation is adapted. So, do not expect any features to run that are not in the test IDL.

Prerequisites

The Ostendo runtime needs Java 1.4 or later, the JUnit test at least Java 1.5 (because of XPathFactory and friends). And of course, you need JacORB-2.2.4.

Download

Download the distribution file ostendo-0.3.1.zip

Additionally you need

  • JacORB 2.3.0 beta 2 from www.jacorb.org
  • xalan-j from xml.apache.org

Install

  • unzip the ostendo distribution
  • copy build.properties.template to build.properties and change the pathes to JacORB and xalan.

Run the demo

In the ostendo folder run ant:

ant rundemo
This executes 
java de.jwi.ostendo.Ostendo ior.txt server.idl \
	messagelog/handleNested-4-Request.bin messagelog/handleNested-4-Reply.bin
(Here's the IDL). This gives the output (here's the handleNested.xml)

The runtime arguments

de.jwi.ostendo.Ostendo ior idl request reply
Ostendo needs the IOR of the server that received the message in order to know the repository ID of the receiving interface. As IIOP messages contain only plain data and no data description, Ostendo needs the corresponding IDL to be able to parse IIOP message. And of course you need the binary messages. You always need at least the request message because only the request contains the operation name which together with the repository ID is needed to look up the operation in the parsed IDL tree.

Grabbing IIOP messages

Normally an Orb does not dump the messages that go over the wire. Therefore I wrote a client interceptor that saves sent and received messages into files. This interceptor relies on JacORB internals and is not portable. Have a look at the demo Client to see how it works.
